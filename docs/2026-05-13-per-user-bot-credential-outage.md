# 2026-05-13 — Per-user bot outage and partial migration

## Short version for Dr. Yoo

The 16 per-user Claude bots (Aixa, Alejandro, Ashley, Axel, Cameron, David, Emily, Jesus, Jose, Lia, Neil, Neil-Claude, Rosi, Stephanie, Zahid, Afrah) all started returning **"Had trouble generating a reply"** around 08:30 PDT. Mskai, Yoo AI, and the three already-migrated bots (Heather, Kaye, Gabriel) were fine.

It's working again now. The 16 bots are back on the **shared-credential model**: each user has `.claude` symlinked to `/home/azureuser/.claude`, each user is in the `azureuser` group, and the master credential is mode 0640 so the group can read it.

**The bots that were already on per-user isolation (Heather, Kaye, Gabriel) are still on per-user isolation.** The 16 others are still on the OLD model. **The per-user isolation migration is NOT finished for them**, and finishing it requires a deliberate interactive step for each account that I cannot do over the run-command proxy.

## What broke

The `/home/azureuser/.claude/.credentials.json` file is the OAuth token that all 16 per-user bots access via a symlink to `/home/azureuser/.claude`. The file was at mode **0600 azureuser:azureuser** — only azureuser could read it. The 16 per-user accounts couldn't read it via the symlink, even though they were in the azureuser group, because of the 0600 mode. The Claude CLI invoked by each bot returned `rc=1: Not logged in · Please run /login`, and the responder fell back to "Had trouble generating a reply".

Why mode 0600 today: the master credential gets refreshed periodically (Claude CLI rewrites it with new OAuth tokens). The CLI's default umask appears to write mode 0600. So every refresh re-breaks the per-user bots until someone re-runs `chmod 0640`.

## What I tried (so the next session doesn't repeat it)

1. **Chmod 0640 + verify in-group** — worked. Cameron and Alejandro produced real Claude replies (10:15, 10:22).
2. **Attempt "finish the per-user isolation migration"** — broke things. Twice. I now know:
   - **`/home/<user>/.claude` is a SYMLINK for 16 users, not a real directory.** `cp + chown + mv` chains followed the symlinks and rewrote the master file. The third user iteration's chown made the master `aixa:aixa`; then I broke it again the second time chowning to `zahid:zahid`. Each fix temporarily knocked mskai offline.
   - **Even if you `unlink` the symlinks and create real per-user `.claude` dirs with copies of azureuser's credential file, the Claude CLI silently fails for the per-user account.** Just the credential file isn't enough — Claude CLI also relies on the device/session state generated by an interactive `claude /login`. heather/kaye/gabriel have that; the 16 don't.
3. **Revert** — current state, working. The 16 per-user `.claude` dirs are back to symlinks, users back in azureuser group, master back to 0640.

## The fragile reality right now — auto-mitigated

- The Claude CLI rewrites `/home/azureuser/.claude/.credentials.json` at mode 0600 when it refreshes the OAuth token. That would normally break the 16 per-user bots each time it happens.
- **An azureuser cron entry installed 2026-05-13 PDT re-applies mode 0640 every minute.** Crontab line:
  ```
  * * * * * /home/azureuser/.claude-cred-chmod.sh >/dev/null 2>&1
  ```
  Helper script at `/home/azureuser/.claude-cred-chmod.sh`. Idempotent — only changes mode if drift detected. Tested: dropping the file to 0600 manually is corrected within 60 seconds.
- Verify with: `crontab -u azureuser -l | grep claude-cred-chmod` and `stat -c '%a' /home/azureuser/.claude/.credentials.json` (should always be 640).
- Worst-case window between refresh and re-chmod: 60 seconds. Per-user bots may post the "Had trouble" fallback for up to a minute after a CLI token refresh, then self-heal.

## Permanent fix — the RIGHT pattern is `CLAUDE_CODE_OAUTH_TOKEN`, not per-user `claude /login`

**Important architectural lesson learned 2026-05-13:** the migration path I attempted (give every bot its own real `.claude` dir + own `.credentials.json` copy) **cannot work** without an interactive browser-based `claude /login` per user. A copy of azureuser's credential file produces `rc=0` with empty stdout — claude CLI silently fails when the credential exists but the device/session state was generated for a different account. Don't repeat this attempt.

**If genuine per-bot isolation becomes a requirement** (e.g., for audit reasons): use the `CLAUDE_CODE_OAUTH_TOKEN` environment variable. The Claude CLI honors it directly — no `.credentials.json` needed, no interactive flow, no token-rotation race. Each bot's systemd service can reference an env var sourced from a long-lived `sk-ant-oat01-...` token in Key Vault, made readable per-bot via per-bot Key Vault Secrets User roles:

```ini
# in /etc/systemd/system/<bot>-responder.service.d/oauth-token.conf
[Service]
Environment=CLAUDE_CODE_OAUTH_TOKEN=$(<read-token-from-keyvault-or-file>)
```

This is the pattern the PACS pipeline uses (`reference_pacs_watchdog_pipeline.md` in user memory). It's the right pattern.

**Bottom line:** the shared-credential-via-symlink model + the per-minute chmod cron is the steady state for the 16. If isolation becomes a hard requirement later, switch to `CLAUDE_CODE_OAUTH_TOKEN` — NOT to per-user `.claude/` with copied credentials.

## Pointer for next sessions

If you see "Had trouble generating a reply" across multiple per-user bots after a credential refresh:

```bash
# Quick fix (restores group-readable on shared credential)
ssh-via-run-command: chmod 0640 /home/azureuser/.claude/.credentials.json
```

**Do NOT** attempt to break the symlinks or chown anything in `/home/<user>/.claude` via `chown -R` — it follows the symlink back to azureuser's directory and corrupts the master.
