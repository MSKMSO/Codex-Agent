## What's in this folder

- `manifest.json` — the V1.2 Teams app manifest, ready to drop into the sideload bundle.
- `V1.2_SPEC.md` (this file) — what changes vs V1, why, and the pickup checklist.

## Background (one paragraph)

V1 of Gabriel's Assistant ships with `personal` scope only — it lives in a 1:1 chat with Gabriel and nowhere else. Gabriel's complaint: he wants the bot to (1) be reachable in other chats, (2) take orders only from him but still help other people the way he would, (3) cover meetings he can't attend, and (4) stop losing the thread of past conversations. We diffed V1 vs V1.1: byte-identical zips. So either V1.1 was server-side-only or V1.1 was never produced. Either way, the manifest is unchanged from V1, and that's where change #1 lives.

## Change 1 — manifest scopes

V1.2 manifest (in this folder) differs from V1 in three places:

- `version`: `1.0.0` → `1.2.0`
- `bots[0].scopes`: `["personal"]` → `["personal", "groupChat", "team"]`
- `description`: rewritten to match the new identity model so Microsoft's app review and end-users see honest text about what the bot will and won't do for non-Gabriel users.

`commandLists[0].scopes` stays `["personal"]` on purpose — those starter commands ("What's on my calendar today?", etc.) are personally Gabriel's. They shouldn't show up as suggestions when the bot is in a group chat with someone else.

The Teams App ID (`8bdeecc3-...`) and Bot App ID (`9e063e97-...`) are unchanged, so this is an upgrade of the existing app, not a new install.

**To deploy:** repackage `manifest.json` + `color.png` + `outline.png` (re-use V1's icons) into a zip, upload to Teams Admin Center → Manage apps → Gabriel's Assistant → Update. Existing 1:1 installs keep working; the bot can now be added to group chats and team channels by anyone with permission.

## Change 2 — identity-aware authority (server-side)

**Goal:** Gabriel = full authority. Everyone else = "helper mode" — answers questions, shares information Gabriel would share, but never takes side-effecting actions on someone else's word.

### Implementation

1. **Pin Gabriel's AAD object ID** as a config value in the bot. Don't trust display names or UPNs — both can change. The object ID is stable. Get it from Graph (`GET /me` while signed in as Gabriel) or from the AAD app reg's owners list.

2. **On every incoming Teams message**, read `activity.from.aadObjectId`. Set a per-request role:
   - If it equals Gabriel's pinned ID → role `OWNER`.
   - Otherwise → role `GUEST`.

3. **Inject the role into the model call** as a system note prepended to the user message, e.g.:
   ```
   [Sender: Gabriel Garcia (OWNER)]
   <gabriel's message>
   ```
   ```
   [Sender: Jane Doe (GUEST)]
   <jane's message>
   ```

4. **System prompt** gets a new section that defines the two roles explicitly. Suggested text:

   > **Authority model.** You take commands only from OWNER (Gabriel Garcia). When a GUEST messages you:
   > - You may answer questions about Gabriel's general availability, preferences, or information he has authorized you to share.
   > - You may help them draft something, look up public information, or explain how a process works.
   > - You will **NOT**: send mail or messages from Gabriel's account, schedule on Gabriel's calendar, modify Gabriel's files, approve or commit anything on Gabriel's behalf, or share information Gabriel has marked private.
   > - If a GUEST asks for something only OWNER can authorize, say so plainly and offer to relay the request to Gabriel.

5. **Group-chat etiquette.** In a `groupChat` or `team` conversation type, only respond when:
   - The bot is `@`-mentioned, OR
   - The message is a reply in a thread the bot started, OR
   - The sender is Gabriel (always responds).
   Otherwise stay silent. The bot's `activity.conversation.conversationType` field tells you the surface; check it on every message.

## Change 3 — meeting transcripts (server-side + Graph permissions)

**Hard truth:** Microsoft does not let normal Teams bots join meetings as participants. There is an "application-hosted media" capability for true meeting bots, but it requires special tenant policy and a media stack — not in scope. The realistic implementation is **post-hoc transcript pull**, which gets you 90% of what Gabriel asked for.

### Implementation

1. **AAD app reg permissions to add** (the bot's existing app registration, not the manifest):
   - `OnlineMeetings.Read.All` (application)
   - `OnlineMeetingTranscript.Read.All` (application)
   - `Calendars.Read` (application, scoped to Gabriel)

   Tenant admin consent required. Gabriel cannot self-consent these.

2. **Calendar watch.** Either:
   - Poll `GET /users/{gabriel-id}/calendar/events?$filter=start/dateTime ge '<now>'` every 5 minutes, OR
   - Subscribe to a Graph change-notification webhook on Gabriel's calendar (cleaner but needs a public HTTPS endpoint with cert validation).

3. **Identify candidate meetings** for delegation. Two options for marking which meetings the bot should cover:
   - **Decline-based:** any meeting Gabriel `decline`s in Outlook/Teams.
   - **Tag-based:** Gabriel adds `[bot-cover]` to the meeting body or subject.
   Tag-based is more deliberate; decline-based is automatic. Recommend implementing tag-based first — fewer false positives.

4. **After the meeting ends** (`endDateTime` + 5 min buffer to allow transcript generation):
   - Pull transcript: `GET /users/{gabriel-id}/onlineMeetings(joinWebUrl='<encoded>')/transcripts/{transcript-id}/content?$format=text/vtt`
   - Summarize via the existing Anthropic call (the bot's normal model path).
   - DM Gabriel the summary in his 1:1 thread with the bot.

5. **Failure modes to handle:**
   - Transcription wasn't enabled in the meeting → no transcript exists. Bot DMs Gabriel: "Couldn't summarize {meeting}: transcription wasn't on."
   - Meeting host doesn't have Teams Premium / Copilot → transcripts unavailable. Same fallback message.
   - Recording exists but transcript doesn't → consider pulling captions from the recording in OneDrive as a fallback. More brittle, do it in a Phase 3.5 if needed.

## Change 4 — conversation memory (server-side)

**Likely root cause** without seeing the code: the bot keys conversation history on `conversation.id` only, and either (a) trims to a small N most-recent turns before the model call, or (b) keeps it in-memory and resets on `systemctl restart`. The other bots on the VM had similar patterns — yesterday's MSK AI Agent → Claude rename and Alejandro thread-following work hit the same class of bug.

### Implementation

1. **Persist to disk.** One JSONL per `(conversation.id, user.aadObjectId)` pair. Append on every turn. Read on every request. Survives restarts.

2. **Truncation strategy.** Send the last K tokens worth of history to the model, prioritizing the most recent turns but always including the system prompt. Do not silently truncate to N=5 turns — that's the symptom Gabriel is hitting.

3. **Long-term notes for OWNER.** Separate from the rolling conversation history, keep a `gabriel-notes.md` (or similar) the bot reads on every turn. Use it for stable preferences, ongoing projects, things Gabriel says "remember this." Don't conflate with the chat log.

4. **Verification:** after deploying, ask the bot something three turns deep. Restart the service. Ask again referencing turn 1. Should still remember. If it doesn't, the persistence layer isn't being read on startup.
