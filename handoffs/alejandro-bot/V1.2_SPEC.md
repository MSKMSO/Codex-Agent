## Change 2 — verify thread-following actually works (server-side)

Before writing code, **test the existing build**:

1. Open a 1:1 chat with the bot. Ask it: "Remember the number 47 for me." It should acknowledge.
2. Send three unrelated follow-up messages.
3. Ask: "What number did I tell you to remember?"

If it answers `47` → thread-following works. Skip to change #3.

If it answers wrong, doesn't remember, or asks again → the persistence layer needs fixing. Likely root cause (same pattern as the other bots on this VM):

- History keyed only on `conversation.id`, not `(conversation.id, user.aadObjectId)`. In a 1:1 those collapse to the same key, but this matters once the bot is in group chats (change #1).
- History trimmed too aggressively before the model call (e.g. last 5 turns).
- In-memory state that resets on `systemctl restart`.

### If a fix is needed

1. **Persist conversation history to disk.** One JSONL file per `(conversation.id, user.aadObjectId)` pair. Append on every turn. Read on every request.
2. **Truncate by tokens, not turn count.** Keep the most recent K tokens of history (target ~60% of the model's context window, leaving room for system prompt + notes file from change #3).
3. **Test for survival across restarts.** Send three messages → `systemctl restart alejandro-bot` → ask the bot what was just discussed. If it remembers, change #2 is done.

## Change 3 — adaptive learning via a per-user notes file

**Plain-English caveat first** so there's no confusion about what this does: the underlying model (Opus 4.7) is fixed and not being trained. What we're building is a **notes file the bot maintains about each user** and reads every time it answers them. It acts like the bot has learned, even though the model is the same.

### Implementation

1. **One notes file per user**, stored at `~/.alejandro-bot/users/<aadObjectId>/notes.md`. Owner's notes are written and read; guests' notes are off-limits in this bot's scope (skip or implement later).

2. **Notes are loaded into the system prompt on every turn.** Cap the file at ~2,000 tokens; if it grows past that, the bot summarizes and rewrites it.

3. **Append triggers — when the bot adds to notes.** Two paths:

   **a) Explicit triggers (start with these):**
   - "Remember [X]"
   - "I always [X]" / "I prefer [X]"
   - "Don't forget [X]"
   - "From now on [X]"
   - "Note: [X]"

   When the bot detects one of these phrases, it extracts the fact and appends to `notes.md` with a date stamp.

   **b) Implicit extraction (Phase 2, optional):**
   A nightly pass that reviews the past 24 hours of OWNER conversations and surfaces patterns it could note ("you mentioned 'osteoarthritis study' six times this week — want me to remember the project context?"). The bot proposes; OWNER confirms before anything is appended. Don't auto-add silently.

4. **Notes file format** (markdown, human-readable):
   ```markdown
   # Alejandro's notes
   _Last updated: 2026-05-02_

   ## Preferences
   - Prefers concise answers, no preamble.
   - Working hours: 8–6 PT, do not page after 7 PM unless urgent.

   ## Active projects
   - **Knee replacement outcomes study** — stats due May 15.

   ## People
   - **Dr. Yoo** — direct supervisor; CC on anything financial.
   - **Gabriel** — coordinator; copy on scheduling.
   ```

5. **OWNER-only commands for managing notes:**
   - "Show me what you remember" → bot pastes the notes file.
   - "Forget [X]" → bot removes the matching line.
   - "Edit your notes" → bot opens an interactive turn where OWNER dictates changes.

### Privacy / safety considerations

- The notes file lives in `~/.alejandro-bot/users/<aadObjectId>/` — same per-bot file scope that yesterday's hardening enforced. Other bots on the VM cannot read it.
- Don't dump the notes file in logs.
- If OWNER asks the bot to "remember my password" or anything credential-shaped, refuse and suggest a password manager. Flag this in the system prompt.
